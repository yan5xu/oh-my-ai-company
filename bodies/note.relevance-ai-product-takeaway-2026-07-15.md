# Relevance AI 产品判断（2026-07-15）
## 一句话判断

[[company.relevance-ai]] 已从无代码 Agent builder 升级为“企业 Specialist Agent 平台 + 嵌入式交付服务”：把 build、runtime、eval、model router、governance、trace 和客户 enablement 放进一个商业闭环。

## 最重要的判断

1. **差异化不只在 no-code。** Evals、发布闸门、运行队列、审批、OTEL 导出和模型路由说明它在争夺 [[concept.agent-lifecycle-control-plane]]。
2. **六周交付是 GTM，不是产品功能。** 先替客户交付第一批 Agent，再训练客户内部专家，是 [[concept.embedded-agent-enablement-loop]]；它能跨过采用障碍，也可能形成服务人力瓶颈。
3. **客户证据有层级。** Send Payments CEO 的客户侧平台确认最强；Qualified/Zembl 数字具体但仍是供应商案例；客户整体运营提升不能全部归因给 Relevance AI。
4. **平台能力与治理仍有硬边界。** prompt injection detection 只观测不阻断；PII redaction 只作用导出；single-tenant 尚非普遍可用；LLM judge 不能替代业务正确性。
5. **GTM 已从自助转向 enterprise sales-led。** 当前官网只谈企业合作，文档与 G2 仍保留 Free/Pro/Team/Business 口径，说明产品与定价正在迁移。
6. **公开流量在下行。** 2026 年 1–6 月主域估算从约 22.7 万/月降到 9.1 万/月；但企业使用可能发生在 app 与内部入口，不能据此判断收入下降。

## 对我们有用的启发

- 企业 Agent 的价值不在“能生成一个 Agent”，而在上线、验证、授权、运行、恢复和持续维护。
- 产品页应把业务 outcome 与 activity metrics 分开；tasks、credits、errors 很重要，但不是业务结果。
- 复杂平台需要一个 adoption loop：首次交付必须有人负责，随后才谈客户自治。
- similar sites 是发现入口，不是竞品结论；Relevance AI 的直接竞品、相邻 workspace、垂直 Agent 和 infra stack 必须分层。

## 待验证

- 真实 ARR/NRR、平均合同、服务收入占比；
- created agents 与 active production agents；
- 六周后客户能否持续自主构建；
- eval false pass、人工接管、故障恢复和 rollback；
- 当前自助套餐是否仍可购买及退款/额度规则。
