# Kernel 产品与增长判断

## 我的 takeaway

1. Kernel 的真正上升路径不是“浏览器启动更快”，而是 browser runtime -> state/auth -> computer use eval -> agent identity。速度是 wedge，认证和委托网络才可能形成长期控制点。
2. 这家公司展示了很强的 founder-market fit：Cash App 的 FDE/商户网站问题给出场景，Clever 的身份基础设施经验给出解法。
3. 第一次有效传播来自具体开源 artifact，而不是品牌故事。HN 上的 Chromium-on-unikernel 讨论远强于后来的泛产品提交；面向开发者时，每次 public launch 都应该像可检验的技术 elevator pitch。
4. GTM 不是只靠 YC 名气。YC/Accel 提供放大，GitHub 和技术文章建立可信度，Luma/DevRel/partnerships 承接社区，客户案例把 infra 转译为业务价值。
5. Website traffic 很容易低估 API-first infra。Kernel 的官网流量小于 Browserbase/Browserless，但单客户月 session、官方 browser launches 和 agents/day 明显更大。规模判断必须双轨。
6. 高价值 browser-agent 场景有共同结构：没有 API、必须登录、站点持续变化、流程受监管或失败成本高。医疗、保险、政府福利比通用网页抓取更能承受认证与 headful 成本。

## 对我们有用的观察

- agent infra 的产品边界正在从“提供工具”转向“提供被信任的执行与身份环境”。
- 可观察性、replay、auth 和权限不是 enterprise 装饰，而是 agent 真正接管长期流程的前提。
- 竞争分析不能只看 SDK feature list；还要看谁掌握网站合作、身份标准、客户认证会话和真实失败数据。

## 仍不下结论

- Kernel 是否有可持续的成本优势。
- Managed Auth 是否能成为独立 moat。
- Web Bot Auth 能否形成跨站点网络效应。
