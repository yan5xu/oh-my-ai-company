# 嵌入式 Agent 交付与客户赋能闭环
企业 Agent 平台常遇到同一个采用断层：买家认同方向，但内部没人能把模糊流程变成可运行 Agent，也没人愿意对第一次失败负责。

[[company.relevance-ai]] 的公开交付路径提供了一种解法：

1. 前 1–2 周与客户共同梳理 use case 和价值；
2. 第 3–6 周由嵌入式交付团队构建首个定制 Agent team；
3. 上线后训练客户内部业务专家，使其能持续构建和运营；
4. 平台把首轮实施经验沉淀为 eval、tool、workflow、policy 和 template。

这不是普通 professional services，也不是完全 self-serve。理想状态下，服务承担启动成本，平台承担后续复制，客户内部 expert 成为扩张节点。

## 判断指标

- 从 use case 到首个生产 Agent 的时间；
- 供应商投入的人天与毛利；
- 六周后客户自主创建/维护比例；
- 首批 Agent 的存活率、扩张数和人工接管；
- 实施经验能否复用为产品能力，而不是每个客户重新定制。

## 风险

如果客户始终依赖供应商，所谓平台会退化为高人力实施业务；如果过早把责任交给客户，又可能因权限、数据、评测和异常恢复不足而失败。

当前信心为 medium：Relevance AI 已明确公开该路径，也有 Send Payments 客户侧部署佐证；尚缺跨客户的服务成本、自主扩张率和长期留存数据。
