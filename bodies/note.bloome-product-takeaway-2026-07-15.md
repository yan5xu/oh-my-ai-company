## 产品判断

1. **Bloome 押注的不是更强单 Agent，而是 Agent 社会的通信容器。** 模型、coding agent 和 runtime 都可以替换，群、关系、共享记忆、权限和协作记录才是产品层长期资产。
2. **“可分享 Agent”让产品和 GTM 合并。** 用户分享群、Agent、Skill 和 artifact 时，也在创建新的获客入口；这比单纯做一个聊天工具再外接营销渠道更有网络复利。
3. **本地 Agent + 云端 Agent + 托管模型的统一成员模型很有价值。** 用户不需要理解背后的 runtime 差异，仍然用同一套 @、回复和线程协作。
4. **ACP 的方向成立，但证据不足以证明结果。** 共享 task state、fresh sensing、output boundary 都是必要条件；是否真的比单 Agent 更好，需要任务成功率、重复率、人工接管率和 token 成本。
5. **Bloome 不是“隐私数据全部本地”。** 消息需要服务端同步，并可能发送给多家模型供应商；真正卖点是可配置运行位置和可见权限，而不是零出端。

## GTM takeaway

- 4 月内测先让熟悉 Agent 的中文创作者做长篇体验，5 月继续扩散教程和真实案例，6 月底再用高完成度视频做官方英文发布，是典型的多阶段 launch。
- 高流量 KOL 内容里，Bloome 有时只是文末 Agent/邀请 CTA，不应把整篇互动都归因于 Bloome；但这说明团队在把 Agent 分享页嵌进创作者内容网络。
- 官网铺了大量 feature/guide 页面，但当前 organic search 只有约 4.6%，SEO 还没有成为主要获客来源。
- 产品大使、Hackathon、邀请链接、Explore 市场和多端 App 共同组成增长网络；目前仍缺留存和付费证据。

## 对 Agent infra 的启发

Bloome 展示了 runtime 与 collaboration surface 的分层：底层可以是本地 Codex、Claude Code 或 E2B sandbox，上层仍用统一成员、线程、权限和交付协议。对 agent infra 创业者而言，竞争不只在“谁能执行”，还在“执行者如何进入团队、被看见、被打断和被复用”。
